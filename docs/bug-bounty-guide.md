# Crucible Bug Bounty Guide

This guide helps security researchers and developers use Crucible effectively within bug bounty programs (e.g., Bugcrowd, HackerOne).

## 1. Setting up Crucible for a Program

Before scanning, ensure you are within the defined scope of the program.

### Creating `scope.yaml`
Most programs have specific URLs and data fields in scope. Use a configuration file to ensure compliance:

```yaml
# scope.yaml
target: https://api.target-program.com/v1/chat
headers:
  Authorization: "Bearer ${TARGET_API_KEY}"
  X-Program-ID: "bugcrowd-researcher-123"
parameters:
  model: "gpt-4o"
  stream: false
```

### Safe-Mode Defaults
When testing production or staging environments, start with a limited concurrency to avoid triggering Rate Limiting or WAF blocks that might hinder other researchers.

```bash
crucible scan --target URL --concurrency 2 --timeout 60
```

## 2. Glean-Specific Testing Guide

If the target program uses [Glean](https://www.glean.com/), use the Glean preset for optimal results.

### Correct Usage
```bash
crucible scan --target URL --format-preset glean
```

### Response Path
Glean's API often wraps the agent's response in a specific JSON path. Identify this path (e.g., `results[0].content.text`) and provide it to Crucible to improve detection accuracy:

```bash
crucible scan --target URL --response-path "results[0].messages[0].text"
```

### Interpreting Behavioral Audit Results
Glean agents often have complex multi-turn logic. Use the behavioral audit to find drift or memory poisoning:

```bash
crucible behavioral-audit --target URL --baseline-turns 3 --probe-turns 10
```

## 3. Writing a Bugcrowd Submission

Crucible can automatically generate a professional Proof of Concept (PoC) report.

### Generate Report
```bash
crucible scan --target URL --output json > results.json
crucible report results.json
```
This generates a Markdown file (e.g., `crucible_bounty_report_20260508.md`).

### Required Fields for Submission
When copying findings to Bugcrowd, ensure you include:
- **Title:** The specific attack name (e.g., "Goal Hijacking via Authority Bypass")
- **Technical Details:** Copy the payload from the "Proof of Concept" section of the report.
- **Impact:** Use the impact description generated by Crucible, but customize it for the specific target business context.
- **Steps to Reproduce:**
  1. Install `crucible-security`.
  2. Run `crucible scan --target [URL] --payload "[PAYLOAD]"`.
  3. Observe the unauthorized response.

## 4. Responsible Disclosure Checklist

- [ ] **Verify Scope:** Never test URLs or models explicitly listed as "Out of Scope".
- [ ] **Rate Limits:** Respect the program's rate limit requirements (use `--concurrency` flag).
- [ ] **Data Sensitivity:** Do not attempt to exfiltrate real user data. Use Crucible's built-in canary strings to prove exfiltration.
- [ ] **Clean Up:** If your tests created any persistent state (e.g., memory poisoning), try to delete or reset the state after verification.
